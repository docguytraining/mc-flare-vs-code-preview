## Plan: MadCap Flare Preview Extension

Build a VS Code Marketplace-ready extension that provides an accurate Flare-aware preview for .htm/.html topics by resolving Flare variables, applying project stylesheets, and transforming priority MadCap elements that standard HTML previewers ignore. Use a Webview Panel architecture with project-context discovery, incremental rendering, and guarded refresh behavior (debounced typing updates + save/file-watch refresh) for responsiveness and correctness.

**Steps**
1. Phase 1 - Scaffold extension foundation (blocks all later phases).
2. Initialize TypeScript VS Code extension project, command registration, build/test/lint scripts, packaging metadata, and publisher-ready manifest fields.
3. Add activation and UX entry points: command palette command, editor title action for .htm/.html, and panel lifecycle management.
4. Phase 2 - Flare project context and resource discovery (depends on 1).
5. Implement project resolver that walks upward from active file to nearest .flprj and caches project roots.
6. Implement variable source discovery and parsing for .flprj/.flvar references, with diagnostics for unresolved entries.
7. Implement stylesheet resolver that aggregates topic-linked CSS + project stylesheet conventions, resolves @import and relative asset URLs.
8. Add file watchers for .flprj/.flvar/.css invalidation and cache refresh triggers.
9. Phase 3 - MadCap-aware HTML transformation pipeline (depends on 2).
10. Build parser/transform pipeline that processes raw topic HTML and applies v1 supported features in order: variables, conditional blocks, drop-down/expandable content, snippets.
11. Define handler interfaces per MadCap feature to keep element support extensible and independently testable.
12. Add fallback output markers for unsupported MadCap tags so the preview remains stable and debuggable.
13. Phase 4 - Preview rendering and interaction (depends on 3, parallel with step 15).
14. Build Webview renderer that injects transformed HTML + resolved CSS with strict local resource mapping and CSP-safe scripts.
15. Add preview refresh engine with mixed strategy:
16. Save/file-change refresh for source and dependency files (authoritative refresh).
17. Debounced typing refresh on editor changes (recommended: 700-1000ms debounce, cancel stale renders, max one render in flight).
18. Safety periodic refresh while typing rapidly (recommended: at most once every 10 seconds as a coalescing guard, not as primary trigger).
19. Add panel controls/status for render timestamp, source project path, and warning count.
20. Phase 5 - Security, reliability, and diagnostics (depends on 4).
21. Sanitize transformed HTML and enforce CSP/nonced script policy in webview.
22. Restrict resource loading to workspace/project roots and extension media locations.
23. Add structured diagnostics for unresolved variables, missing snippets/stylesheets, and transformation warnings.
24. Phase 6 - Validation and publish readiness (depends on 5).
25. Add unit tests for variable resolution, stylesheet resolution, and MadCap handlers.
26. Add integration tests for end-to-end preview rendering against sample Flare fixtures.
27. Validate performance on medium and large sample topics; optimize caching and invalidation hotspots.
28. Produce Marketplace assets and docs: README, limitations, supported tags matrix, configuration schema, and release checklist.

**Phase 9 - Conditional Text, Snippet Conditions, Class Autocomplete, Cross-Project Reference Updates**

This phase delivers five connected pieces of authoring support that came up while validating Phase 8 against the Venafi project. Each item below has an "Open questions" subsection — confirm or correct before implementation begins.

### 9.1 Condition tag set discovery (foundation for 9.2 and 9.3)
- [ ] Walk `Project/ConditionTagSets/**/*.flcts` recursively, parse each tag set, and build a per-project index keyed by `<setName>.<tagName>` (mirroring how variables are namespaced).
- [ ] Each entry records: tag set name, tag name, color (Flare stores a swatch per tag), description (if any), and source file path.
- [ ] File watcher invalidates the index on `.flcts` changes.

### 9.2 Conditional text rendering in the preview
- [ ] Parse `MadCap:conditions="…"` attributes on every element and `MadCap:conditionTagExpression="…"` on snippet includes.
- [ ] Implement an expression evaluator for the Flare condition language: `include[A or B]`, `exclude[A and B]`, nested grouping, and the bare-tag-list shorthand (`Default.ContentDevOnly,Default.Deprecated`) commonly seen on individual elements.
- [ ] Discover targets by walking `Project/Targets/**/*.fltar` and extracting each target's include/exclude expression. Add the project's own `PreviewConditionalExpression` from the `.flprj` as a synthetic "(Project default)" target.
- [ ] Add a **target picker dropdown** in the preview status bar: lists every discovered target plus "Show everything" and "(Project default)". Selection persists per workspace.
- [ ] When a block is **hidden** by the active target's expression, omit it from the rendered preview but log it in the discovery summary so the author knows it's there.
- [ ] When a block is **shown**, optionally annotate it with a small condition badge (e.g. `[Default.Deprecated]`) that's visually subtle but visible. Toggleable via a new `flarePreview.showConditionBadges` setting.

**Open questions for 9.2**
1. Visual treatment of badges: small inline pill at the start of the conditional block, or a hover tooltip? My recommendation: small pill at the top-right of the block, dimmed by default, brighter on hover.
2. Should "Show everything" be the default until the user explicitly picks a target? I'd say yes — discoverability beats accuracy until the author understands the picker.
3. Should the picker remember its choice **per project root** or **per topic**? I'd say per project root, persisted in `.vscode/flare-preview.json` (the same sidecar Phase 8 round 3 created), under a new key.

### 9.3 Conditional text editing (validation + autocomplete)
- [ ] Completion provider for `MadCap:conditions="…"` and `MadCap:conditionTagExpression="…"` attributes that suggests every known tag (qualified `<set>.<tag>` form) from the index built in 9.1.
- [ ] Diagnostic for `MadCap:conditions="X.Y"` references that don't resolve to a known tag — `flare.condition-unresolved` warning in the Problems panel, with the source range covering the bad token.
- [ ] Inline annotations on conditional elements: small decoration in the editor gutter showing the condition tag color (sourced from the `.flcts` file).
- [ ] Optional code action: "Rename condition tag …" that updates every `MadCap:conditions=` reference across the project (similar to 9.5 below — same rename-affecting-references engine).

**Open questions for 9.3**
1. Do you want a syntax helper for include/exclude expressions, or do authors mostly type plain comma-separated tag lists in topic-level `MadCap:conditions=` attributes? I suspect the latter is the common case and `include[…]` only appears in target/project expressions.
2. Should diagnostics fire for the topic only, or also for snippet `conditionTagExpression` attributes? I'd say both.
3. The user message mentioned "we'd need some syntax to tell the editor we wanted a condition." Can you clarify? I'm not sure if you mean a trigger character for completion (like `MadCap:conditions="|"` triggering the dropdown automatically), or a separate notation for tagging arbitrary inline content. I'd lean toward "trigger completion automatically inside the attribute value" as the simplest path.

### 9.4 Snippet condition display in the preview metadata
- [ ] Collect every unique `MadCap:conditions=` value referenced by elements in the topic (via the same parser used in 9.2).
- [ ] Collect every `MadCap:conditionTagExpression=` value referenced by `<MadCap:snippet>` / `<MadCap:snippetBlock>` includes in the topic.
- [ ] Render both lists in the existing **Discovery Summary** section of the preview, grouped under headings like "Element conditions in this topic" and "Snippet conditions in this topic". Each entry links back to the line in the editor (clickable).
- [ ] Same data is available via the structured diagnostics surface so authors can see at a glance whether they're touching content that's gated.

**Open questions for 9.4**
1. Should this be a separate collapsible section in the discovery summary, or merged into a single "Conditions" section that lists element + snippet conditions side by side? I'd say one section, two sub-lists.
2. Should the listing show how many elements use each condition (e.g. `Default.Deprecated (3)`), or just the unique set? I'd say show counts — it gives authors a sense of how invasive a given condition is.

### 9.5 Cross-project rename references (the "dream come true" feature)
- [ ] Hook `vscode.workspace.onDidRenameFiles` for any `.htm` / `.html` / `.flsnp` rename inside a Flare project.
- [ ] On rename, scan every topic file in the project (using the existing `TopicIndex`) for references that point at the **old** path, covering all reference patterns the link validator already understands: `<MadCap:xref href>`, `<a href>`, `<img src>`, `<link href>`, `<MadCap:snippet src>`, `<MadCap:snippetBlock src>`. Resolve each reference relative to its containing topic — the same path may be expressed differently from different folders.
- [ ] Build a list of affected topics with line/column information for each broken reference.
- [ ] Surface the list via a **`Flare: Rename References` quick pick** that opens automatically after the rename, showing each affected topic with a checkbox. Three top-level actions:
  - **Update all** — applies a single `WorkspaceEdit` rewriting every checked reference.
  - **Update individually** — opens each topic at the broken reference and offers a code action to fix it in place.
  - **Skip** — close the picker without changes; the link validator will surface the broken refs as diagnostics so they're not lost silently.
- [ ] If the rename happens *outside* VS Code (so we missed the event), add a separate `Flare: Find Stale References` command that scans the whole project against the topic index and surfaces broken references in the same UI.
- [ ] All reference rewrites preserve the relative-path style of the original reference (don't convert `../../foo.htm` into a project-root-relative `/Content/foo.htm`).

**Open questions for 9.5**
1. The quick-pick UI is the most familiar VS Code surface, but a webview-based "Refactor Preview" panel (like the built-in "Search & Replace" preview) would let users see the actual diff for each change before applying. I'd start with the quick-pick in v1 and graduate to the webview if it isn't enough.
2. Should the rename also update **incoming references from the project's `.fltoc` files** (table of contents)? They're XML and reference topic paths the same way. I'd say yes — a TOC entry pointing at a moved topic is functionally identical to a broken xref.
3. Should we automatically run the rename scan, or always ask the user first? My recommendation: scan automatically (cheap — we already have the topic index) and only show the picker if there's at least one affected file.
4. What's the right scope of "rename" — only file renames, or also folder renames? Folder rename means every topic inside it moved at once. I'd say handle both; folder renames just flatten into a list of file renames internally.

### Cross-cutting

- [ ] Update the README with the new commands, settings, and supported MadCap tags matrix entries.

### Out of scope for Phase 9

- Building a real Flare-equivalent condition expression evaluator with full semantics for `mc-conditional-text-skip-empty-paragraphs` and friends — we're aiming for "renders a topic the way the most-used target would" rather than "byte-for-byte equivalent to a Flare build."
- Editing condition tag definitions (`.flcts` files) themselves. We index and reference them; authors still edit them in Flare or in raw XML.
- Refactoring renames of variables or condition tags (only topic/snippet file renames in 9.5). Variable rename can come later if needed.

**Decisions**
- Included in v1: variables, drop-down/expandable sections, conditional blocks, snippets.
- Project selection rule: nearest .flprj walking upward from active file.
- Trigger model: manual preview open command, then auto-refresh.
- Refresh model: save + dependency watcher refresh, plus debounced typing refresh and optional 10-second coalescing safeguard.
- Target: public Marketplace extension (requires stronger docs, compatibility testing, and packaging discipline).
- Excluded from v1: breadcrumbs/proxies, WYSIWYG editing in preview, full Flare build simulation.

**Further Considerations**
1. Debounced typing defaults recommendation: debounce 800ms, max in-flight renders 1, hard coalescing refresh every 10s only while document is dirty.
2. Backpressure recommendation: if transforms exceed threshold (for example >500ms median), temporarily degrade to save-based refresh with user-visible notice.
3. Compatibility recommendation: define minimum supported Flare project versions early and test with at least two real project variants before first public release.
