## Plan: MadCap Flare Preview Extension

Build a VS Code Marketplace-ready extension that provides an accurate Flare-aware preview for .htm/.html topics by resolving Flare variables, applying project stylesheets, and transforming priority MadCap elements that standard HTML previewers ignore. Use a Webview Panel architecture with project-context discovery, incremental rendering, and guarded refresh behavior (debounced typing updates + save/file-watch refresh) for responsiveness and correctness.

**Steps**
1. Phase 1 - Scaffold extension foundation (blocks all later phases).
2. Initialize TypeScript VS Code extension project, command registration, build/test/lint scripts, packaging metadata, and publisher-ready manifest fields.
3. Add activation and UX entry points: command palette command, editor title action for .htm/.html, and panel lifecycle management.
4. Phase 2 - Flare project context and resource discovery (depends on 1).
5. Implement project resolver that walks upward from active file to nearest .flprj and caches project roots.
6. Implement variable source discovery and parsing for .flprj/.flvar references, with diagnostics for unresolved entries.
7. Implement stylesheet resolver that aggregates topic-linked CSS + project stylesheet conventions, resolves @import and relative asset URLs.
8. Add file watchers for .flprj/.flvar/.css invalidation and cache refresh triggers.
9. Phase 3 - MadCap-aware HTML transformation pipeline (depends on 2).
10. Build parser/transform pipeline that processes raw topic HTML and applies v1 supported features in order: variables, conditional blocks, drop-down/expandable content, snippets.
11. Define handler interfaces per MadCap feature to keep element support extensible and independently testable.
12. Add fallback output markers for unsupported MadCap tags so the preview remains stable and debuggable.
13. Phase 4 - Preview rendering and interaction (depends on 3, parallel with step 15).
14. Build Webview renderer that injects transformed HTML + resolved CSS with strict local resource mapping and CSP-safe scripts.
15. Add preview refresh engine with mixed strategy:
16. Save/file-change refresh for source and dependency files (authoritative refresh).
17. Debounced typing refresh on editor changes (recommended: 700-1000ms debounce, cancel stale renders, max one render in flight).
18. Safety periodic refresh while typing rapidly (recommended: at most once every 10 seconds as a coalescing guard, not as primary trigger).
19. Add panel controls/status for render timestamp, source project path, and warning count.
20. Phase 5 - Security, reliability, and diagnostics (depends on 4).
21. Sanitize transformed HTML and enforce CSP/nonced script policy in webview.
22. Restrict resource loading to workspace/project roots and extension media locations.
23. Add structured diagnostics for unresolved variables, missing snippets/stylesheets, and transformation warnings.
24. Phase 6 - Validation and publish readiness (depends on 5).
25. Add unit tests for variable resolution, stylesheet resolution, and MadCap handlers.
26. Add integration tests for end-to-end preview rendering against sample Flare fixtures.
27. Validate performance on medium and large sample topics; optimize caching and invalidation hotspots.
28. Produce Marketplace assets and docs: README, limitations, supported tags matrix, configuration schema, and release checklist.

**Phase 9 - Conditional Text, Snippet Conditions, Class Autocomplete, Cross-Project Reference Updates**

This phase delivers five connected pieces of authoring support that came up while validating Phase 8 against the Venafi project. Each item below has an "Open questions" subsection — confirm or correct before implementation begins.

### 9.1 Condition tag set discovery (foundation for 9.2 and 9.3)

- [x] Each entry records: tag set name, tag name, color (Flare stores a swatch per tag), description (if any), and source file path.
- [x] File watcher invalidates the index on `.flcts` changes.

### 9.2 Conditional text rendering in the preview
- [x] Parse `MadCap:conditions="…"` attributes on every element and `MadCap:conditionTagExpression="…"` on snippet includes.
- [x] Implement an expression evaluator for the Flare condition language: `include[A or B]`, `exclude[A and B]`, nested grouping, and the bare-tag-list shorthand (`Default.ContentDevOnly,Default.Deprecated`) commonly seen on individual elements.
- [x] Discover targets by walking `Project/Targets/**/*.fltar` and extracting each target's include/exclude expression. Add the project's own `PreviewConditionalExpression` from the `.flprj` as a synthetic "(Project default)" target.
- [x] Add a **target picker dropdown** in the preview status bar: lists every discovered target plus "Show everything" and "(Project default)". Selection persists per workspace.
- [x] When a block is **hidden** by the active target's expression, omit it from the rendered preview but log it in the discovery summary so the author knows it's there.
- [x] When a block is **shown**, optionally annotate it with a small condition badge (e.g. `[Default.Deprecated]`) that's visually subtle but visible. Toggleable via a new `flarePreview.showConditionBadges` setting.

**Decisions for 9.2** *(confirmed)*
1. **Badges:** small pill at the top-right of each conditional block, dimmed by default, brighter on hover. The pill text is the qualified `<set>.<tag>` name.
2. **Default target:** "Show everything" until the author explicitly picks a target. Discoverability first.
3. **Picker persistence:** per project root, stored in `<projectRoot>/.vscode/flare-preview.json` under a new top-level key (e.g. `previewTarget`). Same sidecar Phase 8 round 3 created — keep one config file per project.

### 9.3 Conditional text editing (validation + autocomplete)
- [x] Completion provider for `MadCap:conditions="…"` and `MadCap:conditionTagExpression="…"` attributes that suggests every known tag (qualified `<set>.<tag>` form) from the index built in 9.1.
- [x] Diagnostic for `MadCap:conditions="X.Y"` references that don't resolve to a known tag — `flare.condition-unresolved` warning in the Problems panel, with the source range covering the bad token.
- [ ] *(Deferred)* Inline gutter decoration showing each condition tag's color from the `.flcts` file. Not implemented in this round.
- [ ] *(Deferred)* "Rename condition tag …" code action that updates every reference across the project. Not implemented in this round.

**Decisions for 9.3** *(confirmed)*
1. **No syntax helper** for `include[…]` / `exclude[…]` in topic-level attributes. Topic `MadCap:conditions=` is plain comma-separated tag lists in practice; full expression syntax only appears in `.fltar` target files and the `.flprj` `PreviewConditionalExpression`.
2. **Diagnostics fire for both** topic-element `MadCap:conditions=` and snippet-include `MadCap:conditionTagExpression=` attributes.
3. **Completion trigger:** automatic, inside the attribute value. Cursor inside `MadCap:conditions="|"` (or after a comma, for adding additional tags) opens the dropdown of every known tag from the project's `.flcts` files. No new author-facing syntax to learn.

### 9.5 Cross-project rename references (the "dream come true" feature)
- [x] Hook `vscode.workspace.onDidRenameFiles` for any rename of an HTML topic, snippet, TOC, alias file, browse sequence, master page, glossary, relationship table, condition tag set, target, page layout, skin, image, font, or PDF — anything Flare can reference. Trigger set lives in `RENAME_TRIGGER_EXTENSIONS` plus the `^\.fl[a-z0-9]+$` family pattern so future Flare project file types are picked up automatically.
- [x] On rename, scan every text-bearing project file (any `.fl*` extension plus `.htm`/`.html`) for references that point at the old path. Uses a single generic attribute pattern (`href`, `src`, `source`, `Link`, `xlink:href`, `File`, `Topic`) so references in TOCs, snippets, alias files, browse sequences, and any future Flare project file type are caught without enumerating container elements.
- [x] Build a list of affected files with line/column information for each broken reference.
- [x] Surface the list via a **`Flare: Rename References` quick pick** that opens automatically after the rename. Each entry has a checkbox (all checked by default), shows the file, line/column, and a `before → after` preview. Pressing Enter applies a single `WorkspaceEdit` rewriting every checked reference.
- [x] `Flare: Find Stale References` command for the case where the rename happened outside VS Code. Scans the project and surfaces stale references in the Problems panel.
- [x] Reference rewrites preserve the relative-path style of the original — project-root-relative refs stay project-root-relative; sibling-relative refs stay relative.

**Decisions for 9.5** *(confirmed)*
1. **Quick-pick UI in v1.** Each affected topic gets a checkbox with line/column info and a preview of the existing reference. Graduate to a webview "Refactor Preview" panel only if quick-pick proves insufficient.
2. **Update `.fltoc` files** too. The rename scanner walks `Project/TOCs/**/*.fltoc` alongside topic files; TOC entries pointing at the moved topic are rewritten with the same logic.
3. **Scan automatically** on every in-IDE rename. The picker only opens if there's at least one affected reference. Silent no-op when nothing references the renamed file.
4. **Handle file and folder renames.** Folder renames flatten internally into a batch of file renames so the same code path serves both. Folder renames may produce a large affected-file list; keep the picker scrollable and add a top-level "Update all" / "Skip all" affordance.

- Building a real Flare-equivalent condition expression evaluator with full semantics for `mc-conditional-text-skip-empty-paragraphs` and friends — we're aiming for "renders a topic the way the most-used target would" rather than "byte-for-byte equivalent to a Flare build."

**Verification**
1. Functional trigger checks: open a Flare topic, run preview command, verify panel opens and resolves nearest .flprj.
2. Variable rendering checks: validate known variable placeholders render expected values and unresolved keys emit diagnostics.
3. MadCap v1 checks: confirm conditional blocks, drop-down/expandable sections, and snippets render correctly in preview.
4. Stylesheet checks: confirm topic and project CSS are applied, including @import and relative assets.
5. Refresh checks: verify save-based refresh, dependency-file watcher refresh, and debounced typing refresh behavior under rapid edits.
6. Security checks: ensure scripts in source are blocked/sanitized and no external unapproved resources load.
7. Scale checks: benchmark render latency and memory on representative larger Flare topics with many variables/snippets.
8. Publish checks: run extension packaging validation and install VSIX in clean VS Code profile.

**Decisions**
- Included in v1: variables, drop-down/expandable sections, conditional blocks, snippets.
- Project selection rule: nearest .flprj walking upward from active file.
- Trigger model: manual preview open command, then auto-refresh.
- Refresh model: save + dependency watcher refresh, plus debounced typing refresh and optional 10-second coalescing safeguard.
- Target: public Marketplace extension (requires stronger docs, compatibility testing, and packaging discipline).
- Excluded from v1: breadcrumbs/proxies, WYSIWYG editing in preview, full Flare build simulation.

**Further Considerations**
1. Debounced typing defaults recommendation: debounce 800ms, max in-flight renders 1, hard coalescing refresh every 10s only while document is dirty.
2. Backpressure recommendation: if transforms exceed threshold (for example >500ms median), temporarily degrade to save-based refresh with user-visible notice.
3. Compatibility recommendation: define minimum supported Flare project versions early and test with at least two real project variants before first public release.
